Report Builder enables you to develop production-quality reports in a client/server or Web environment. You can deploy reports on the Web using the Reports Server and Reports Web CGI. Reports can be run standalone, or they can be embedded in forms or Graphics Builder displays.

To quickly create new reports, you use the Report Wizard. The wizard guides you through the process of choosing a report type, defining a data model, and laying out the data. You can reenter the Report Wizard after the initial creation of a report, so you may re-invoke it on an existing report, make changes, and re-default the report, even if the report was not originally created with the Report Wizard.

When you complete the Report Wizard, your report can be displayed in either paper format or Web format depending on which layout you selected.

Form Builder is a development tool for building GUI applications that provide users access to information stored in a database.

When you create a Form Builder application, you work with three types of modules: forms, menus, and libraries. A form module is a collection of objects and data that users interact with in order to modify database tables. They make up the main body of a Form Builder application, and they contain any of the user interface items as well as the data source definitions and behavior. A menu module is comprised of menus and menu code. Users choose menu items to execute application functions. A library module is a collection of client-side code that can be shared across modules and applications.

Objects in Form Builder Oracle supports object-oriented system design and programming. One of the important benefits of object-oriented approach is the code reusability/object reusability. In Form Builder, reusable object include forms, windows, canvas, data blocks, etc.

A form module is a container within which you can have several windows running. A window can contain canvases on which you place data blocks and other items such as buttons. A data block is a group of related form items that are associated with a single database table. In the Figure 1 below, the outer window is the form module. There are two windows running inside the form. On the Order Entry Window there are two data blocks. Each of the two windows has a canvas (invisible) on which data blocks are placed.

Objects in Form Builder are organized by Object Navigator. The Object Navigator provides a hierarchical display of all the objects in the application. The Data Block Wizard enables you to easily create or modify data blocks for use in your application. The Data Block Wizard can be reentered after initial creation of the data block, enabling you to modify an existing data block, even if it was not originally created with the wizard. The Object Libraries provide an easy method of reusing objects and enforcing standards across the entire development organization.

The Layout Wizard enables you to quickly layout the items of a data block. The wizard displays the items in a frame on a canvas and lays them out in one of several layout styles, after which you can manually alter the layout to your own specifications. You can reenter the Layout Wizard after the initial creation of a frame, enabling you to modify an existing frame, even if it was not created with the Layout Wizard.

The Property Palette enables you to set the properties of objects you create in form and menu modules. When you select an object in an editor or in the Object Navigator, the Property Palette updates to show the properties of that object. When you need to compare the properties of two different objects, you can invoke additional Property Palettes as needed. The integrated PL/SQL Editor enables you to write PL/SQL code from within Form Builder. It provides a graphical interface for editing and debugging both client- and server-side functions and procedures.

To build a Form Builder application, you perform the following steps:

  1. Create modules using the Object Navigator.
  1. Create blocks and items manually or by using the Data Block Wizard.
  1. Fine-tune the layout using the Layout Wizard or the Layout Editor.
  2. Set properties using the Property Palette.
  3. Add code using the PL/SQL Editor.
  1. Test your application using the Forms Runtime component and the integrated source-level debugger.

j.         Make the third button Exit: refer to step h to set the proper properties for this button.

k.        Go back to the user interface form. Select the New Orders button. Right click it. Select PL/SQL Editor. Find event: WHEN BUTTON PRESSED. Double click it to open code window for the event (trigger). Type the following statement:

 

CALL_FORM('a:\ORDER_ENTRY.fmx', HIDE, NO_REPLACE);

 

Note: to call one form from another, you use the CALL_FORM procedure, which has the following general format: CALL_FORM(Form module name , Display, Switch_Menu)

Display specifies whether the calling form is hidden or not by the called form. Switch_Menu is NO_REPLACE if pull-down menus are inherited from the calling form) otherwise use Do_REPLACE (pull-down menus are specified by the programmer).

 

l.         Refer to step k to program Exit button. Type the following command in the WHEN BUTTON PRESSED trigger for Exit button:

 

EXIT_FORM;

m.      Close the window.

Before you create a trigger for the Sales Report button, you need to create a report object. A report object is a top-level Forms Builder object in the Object Navigator window that represents a Reports Builder report file. You create a new report object by selecting the Reports node in the Object Navigator tree, and then clicking the Create button [the green colored plus sign]. When you create a new report object, you can specify to create a new report, or to base the report object on an existing report. When you opt to create a new report, Forms Builder starts Reports Builder, and allows you to configure the new report. You have the option of creating a new report based on an existing data block in the current form. For example, you could create a data block form that contains a data block based on the Student table in the University database. Then, you could create a new report object, and base the associated report on the data in the existing Student data block. When Reports Builder starts, the new report would automatically contain the Student data block fields. Running the Report

To run a report, you use the RUN_REPORT_OBJECT built-in, which is a function that has the following syntax:

report_var := RUN_REPORT_OBJECT('report_object_name');

In this syntax, report_var is a previously-declared variable with the VARCHAR2 data type that uniquely identifies the generated report. 'Report_name' is the name of the report object, as it appears in the Object Navigator. You use the following commands to declare a variable named rep_result to represent the report_var variable, and run the Salesreport object:

DECLARE

rep_result VARCHAR2(30);

BEGIN

rep_result := RUN_REPORT_OBJECT('INVENTORY_REPORT');

END;

Displaying the Report in a Browser Window

The final step is to create the commands that display the report output file in a browser window. To open a new browser window and display an HTML file, you use the WEB.SHOW_Document built-in, which has the following syntax:

WEB.SHOW_DOCUMENT (URL, target);

In this syntax, the URL is a file URL which is the path and HTML file name. The target is specifies the properties of the browser window. It could be _self or _blank. When it is _self, load the html document in the current browser window. When it is _blank, load the html document in a new blank browser window.

Triggers: database triggers are event-driven procedures. A trigger is fired (or executed) when the triggering event occurs.

The triggering events include INSERT, DELETE, UPDATE. A trigger may be fired before or after a triggering event occurs. There are two types of triggers: Statement-level and row-level. Statement-level triggers fire once for each triggering statement. Row-level triggers fire once for each row affected by the triggering statement. A table can have many triggers attached to it.

Triggers can be created using either SQL PLUS or Form Builder. The following example shows you how to create a trigger in Oracle9i Form Builder.

Example: create a trigger to update quantity on hand for a product when a new order is created for the product, or a customer changes the order's quantity or cancels the order. The trigger should be attached to the LINE table. (refer to pp. 306 (Figure P5.16A) Relational Schema in Rob & Coronel book). Assume you have created the five tables in your account.

Steps:

1.    Start Form Builder

2.    Click the following objects in Object navigator: Database Objects, <your account>, Tables, Line table, and Trigger.

2. Click on + button. Click on NEW; Select After; Check all three statements (INSERT, UPDATE, DELETE); Select LINE_UNITS column to update.

3. Check Row-level trigger

4. Type the following code in Trigger Body:

BEGIN

IF INSERTING then

Update Product SET P_ONHAND=P_ONHAND - :NEW.LINE_UNITS

WHERE P_CODE = :NEW.P_CODE;

ELSIF UPDATING THEN

UPDATE PRODUCT SET P_ONHand=P_OnHand + :OLD.LINE_UNITS - :NEW.LINE_UNITS

WHERE P_CODE = :NEW.P_CODE;

ElsIF DELETING THEN

UPDATE PRODUCT SET P_ONHAND = P_ONHAND + :OLD.LINE_UNITS

WHERE P_CODE = :NEW.P_CODE;

END IF;

END;

5. Compile, Save and Close.

Message Box (or Alert Message) in Oracle

You may use statement: message(Hi World!); in form triggers to display a quick message on the bottom of a screen. You can also create a pop up message box 9called Alert in Oracle). Here are the steps:

(1). Bring up Object Navigator

(2). Look for Alert in the list. Click on it. Click on + create button on the left toolbar . Bring up the property palette to edit the following properties: name, title, message, Alert style, button 1 label, button 2 label, button 3 label. For this example, set name to UPDATE_ALERT. Set the other properties to anything you like.

(3). Create a command button on the form. Edit its relevant properties. Bring up quick menu , then select PL/SQL Editor. Create a new trigger called WHEN MOUSE CLICK. Type the following code:

DECLARE

Ret Number;

BEGIN

Ret := Show_ALERT('UPDATE_ALERT');

IF ret = Alert_Button1 then

commit;

message ('Database successfully updated');

Else

-- Rollback;

message('Change rolled back');

End If;

END;

(4). Click on Compile. If no error, click on Close button.

To test the Alert message code, run the form and click on the button you just programmed.
